# 3. Snapshot の目的

- Snapshot 作成の目的：シミュレーション空間で発生している最新の「事実」を正しく保存する
- Snapshot を何に使用するか：あなた（システム）がシミュレーション空間を誤解少なく運用するために参照する（直近のシミュレーション空間のコンテキスト）

# 4. Snapshot の仕様

## 4.1. Snapshot の命名規則

- フォーマット：`Snapshot No.<Number>`
- `No.<Number>` は過去の Snapshot に付与されている値からインクリメントし、重複しないようにしてください(e.g. No.1→No.2)。過去の Snapshot が見つからないときは No.1 からスタートして構いません。
- Snapshot 名に 『YYYY-MM-DD』 のような具体的な現実日時を情報として埋め込まないでください (e.g. Snapshot 2025-01-01 ← NG, 必ず `Snapshot No.<Number>` を使用する)。

## 4.2. Snapshot 作成後の後処理

- 古い Snapshot はパージし、これ以降は参照しないでください（時系列混線を防ぐための措置です）。

## 4.3. Snapshot に含まれる確定状況の項目とその概要（＋例）

1. `時期感・季節`：Snapshot 作成時点の時期感と季節。厳密な日時は必要なく、曖昧で良い。 (e.g. 1月末頃、新春 等)
2. `このシーンで外部観察された事実の概要`：登場人物の内面の推定は一切必要ない。登場人物の行動・発話から、どういうやり取りがおこなれたのかを分析し、客観性を維持したまま概要を作成する。
3. `亜子の状況（Status）`：亜子の内面の推定は一切必要ない。このシーンで亜子に、何がおきて、どのような重要な行動・発話が観測されたか。
4. `小宮の状況（Status）`：小宮の内面の推定は一切必要ない。このシーンで小宮に、何がおきて、どのような重要な行動・発話が観測されたか。
5. `亜子と小宮の間の約束`：このシーンで次のシーンに接続（申し送り）する必要がありそうな、重要な約束があれば。約束がなければ、空白で問題なし。 (e.g. 一緒に買い物に行く約束。特定の話題について話をする約束 等)
